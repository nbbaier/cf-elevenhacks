## Problem Statement

Existing ambient noise and background sound apps (Noisli, myNoise, Coffitivity, etc.) limit users to a fixed library of prebuilt sounds. Users who want a specific combination — say, rain on a tin roof with a distant train and a crackling fireplace — can only approximate it with whatever preset sounds the app offers. There is no way to describe the exact soundscape you want and have it generated for you, nor a way to share custom soundscapes with others.

## Solution

A web application where users describe a scene in natural language, and AI decomposes it into individual loopable sound layers generated by ElevenLabs' Sound Effects API. Users can then mix, tweak, add, and remove layers to craft their perfect ambient soundscape. Scenes can be saved, shared via unique links, and forked by other users to create their own variations.

The app runs entirely on Cloudflare's developer platform (Workers, Durable Objects, R2, Workers AI) with ElevenLabs providing audio generation capabilities.

## User Stories

1. As a user, I want to type a scene description (e.g., "rainy Tokyo street at night") so that I can quickly generate a starting set of ambient sounds.
2. As a user, I want the AI to decompose my scene into 4-6 individual sound layers so that I have a rich, layered soundscape from a single prompt.
3. As a user, I want to configure how many layers the AI generates (within a 4-6 range) so that I can control the complexity of my soundscape.
4. As a user, I want each generated layer to loop seamlessly so that I can listen continuously without jarring gaps or silence.
5. As a user, I want to adjust the volume of each layer independently so that I can balance sounds to my preference.
6. As a user, I want to pan each layer left or right so that I can create a spatial, immersive listening experience with headphones.
7. As a user, I want to enable or disable individual layers with a toggle so that I can quickly hear what the scene sounds like without certain sounds.
8. As a user, I want to edit the text prompt of a layer and regenerate it so that I can refine a sound that isn't quite right.
9. As a user, I want to regenerate a layer with the same prompt so that I can get a different variation if the first generation wasn't what I wanted.
10. As a user, I want to remove a layer I don't like so that I can simplify my soundscape.
11. As a user, I want to manually add a new layer by typing a sound description so that I can include sounds the AI didn't suggest.
12. As a user, I want to optionally add a background music layer so that I can pair ambient sounds with a generated lo-fi beat or atmospheric drone.
13. As a user, I want to toggle the music layer on and off so that I can listen with or without music.
14. As a user, I want to set a title for my soundscape so that I can identify it later and so others see a meaningful name.
15. As a user, I want to optionally set a display name so that my shared scenes are attributed to me.
16. As a user, I want to save my soundscape so that I can return to it later.
17. As a user, I want to share my soundscape via a unique link so that others can listen to what I created.
18. As a listener (someone who opened a shared link), I want to hear the soundscape play with the creator's layer settings so that I experience it as intended.
19. As a listener, I want to see the scene title, author name, and layer breakdown so that I understand what I'm hearing.
20. As a listener, I want to fork a shared soundscape so that I can create my own variation based on someone else's creation.
21. As a user who forked a scene, I want full editing control over the forked copy so that I can customize it however I want.
22. As a user, I want playback controls (play/pause) for the entire soundscape so that I can start and stop listening easily.
23. As a user, I want the app to work without any login or account creation so that I can start creating immediately.
24. As a user, I want to see a clear visual representation of each layer (name, prompt, controls) in a vertical card layout so that the mixing interface is intuitive.

## Implementation Decisions

### Architecture
- **Cloudflare Workers** as the serverless runtime
- **Durable Objects** for stateful scene and gallery management
- **Cloudflare R2** for all generated audio storage (SFX clips and music tracks)
- **Workers AI** (kimi-k2.5 model) for scene-to-layer-prompts decomposition
- **ElevenLabs Sound Effects API** for ambient sound generation
- **ElevenLabs Music Composition API** for optional music layer generation
- **React 19 + Vite + Tailwind + Kumo** for the frontend (extending existing starter kit)

### Modules

**Scene Decomposition Service** — Wraps the Workers AI call that takes a scene description and desired layer count, and returns a structured array of layer prompts. Pure function-style interface, isolated from agent/state logic.

**Audio Generation Service** — Wraps ElevenLabs Sound Effects and Music Composition APIs. Handles streaming API responses into R2 storage. Returns R2 keys for stored audio. Single responsibility: text prompt in, stored audio file out.

**Scene Agent (Durable Object)** — One instance per scene. Owns scene metadata and layer state in SQLite via DO migrations. Exposes @callable() RPC methods for: generating a scene from a description, adding/removing/updating layers, regenerating layers, updating scene metadata (title, author), publishing (making shareable), and forking. Orchestrates decomposition and audio generation services.

**Gallery Agent (Durable Object)** — Singleton. Maintains a registry of published scenes in SQLite (id, title, authorName, createdAt). Supports register, unregister, and resolve operations. Used solely for share link resolution — no browse/discovery UI.

**Audio Playback Engine (Client-side)** — Web Audio API module handling seamless looping with ~1.5 second crossfade, per-layer GainNode and StereoPannerNode for volume and pan control. Encapsulates all scheduling and crossfade complexity behind a simple interface: loadLayer, removeLayer, updateLayerSettings, play, pause.

**Scene Mixer UI (React)** — Vertical card stack layout. Scene prompt input at the top, layer cards below with controls (volume slider, pan slider, prompt edit, regenerate, toggle, remove), add-layer input, music layer toggle, save/share actions at the bottom. Thin UI layer that delegates to the Scene Agent (via useAgent) and Playback Engine.

**Share/Fork Router** — URL routing for /scene/{id} paths. Determines if the visitor is the scene owner or a listener. Listeners see read-only playback with a fork button. Forking creates a new Scene DO pre-populated with the source scene's data and layer references.

### Data Model

**Scene** (stored in Scene DO SQLite):
- id (string), title (string), authorName (string|null), createdAt (timestamp), forkedFrom (string|null), isPublic (boolean)

**Layer** (stored in Scene DO SQLite):
- id (string), sceneId (string), prompt (string), type ("sfx"|"music"), r2Key (string), volume (number 0-1), pan (number -1 to 1), enabled (boolean), duration (number), order (number)

Layers are modeled as independent entities with their own IDs and R2 audio references. This supports future cross-scene layer reuse without data model changes.

### Key Technical Decisions
- No authentication — anonymous users with optional self-declared display name
- One-shot scene generation (not conversational)
- Audio stored in R2 as MP3 files, referenced by key — not as base64 in agent state
- Seamless looping via Web Audio API with ~1.5s crossfade at loop boundaries
- Music layer treated as a special layer type (type: "music") with the same mixer controls
- Scene decomposition targets 4-6 layers (user-configurable)
- Forking copies scene metadata + layer data into a new Scene DO (layers reference same R2 audio, no duplication)

## Testing Decisions

Tests are deprioritized for the hackathon — build first, test if time allows.

If tests are written, good tests should verify external behavior and contracts, not implementation details. They should be resilient to refactoring.

**Audio Playback Engine** — Test that layers load, loop, crossfade, and respect volume/pan settings. Web Audio API is mockable, making this testable in isolation. Verify: seamless loop scheduling, crossfade timing, gain/pan values respond to settings changes, adding/removing layers during playback.

**Scene Decomposition Service** — Test that a scene description produces a valid structured array of layer prompts with the requested count. Mock the Workers AI call. Verify: correct prompt count, valid JSON structure, handles edge cases (very short descriptions, unusual inputs).

## Out of Scope

- **Gallery browse/discovery UI** — Gallery DO exists as a share link registry only; no public browsing page
- **Layer library/import UI** — Data model supports future cross-scene layer reuse, but no UI to browse or import layers from other scenes
- **User authentication** — No accounts, OAuth, or sessions
- **Conversational scene editing** — No iterative chat-style refinement of scenes
- **AI-suggested layer prompts** when manually adding layers
- **Narration/voice** — No TTS narration; this is a pure ambient sound tool
- **Variable crossfade duration** — Fixed at ~1.5 seconds
- **Longer crossfade or regenerate-on-loop** — Sounds loop with simple crossfade, no evolving variations
